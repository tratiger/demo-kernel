#  デモカーネル拡張・設計ルール

## 1. ディレクトリ構造とレイヤリング原則

カーネルは、以下の5つのレイヤで構成し、厳格な階層化を守る。

### 【ルール1-1】依存方向の単一化（上層から下層への依存）

* 上位レイヤ（例: `kernel/`）は、下位レイヤ（例: `drivers/` や `mm/`）を呼び出して良い。
* **下位レイヤ（例: `arch/`, `mm/`）は、上位レイヤ（例: `fs/`, `kernel/`）の存在を知ってはならない（`use` してはならない）**。
* 同一レイヤ内での横方向の依存は、結合度を最小限にする（例: `serial` が `keyboard` を参照しない）。

### 【ルール1-2】新規コンポーネントの配置ルール

新しい機能を追加する際は、フラットに置くことを禁止し、以下の分類に従って必ず深い階層を作る。

* **CPU/チップセット固有の制御** ➔ `src/arch/`
* **論理的なメモリ操作・管理** ➔ `src/mm/`
* **ファイルシステムの共通ルールの定義やパス解決** ➔ `src/fs/`
* **スレッドやタスク制御の中核** ➔ `src/kernel/`
* **上記以外のすべての具象実装（ハードウェア制御、個別のFSなど）** ➔ `src/drivers/` 配下の適切なサブディレクトリ（`char/`, `block/`, `fs/` など）

## 2. ファイル分割（3層分離）の厳格なルール

1つのコンポーネント、モジュールを設計する際は、機械的に、型（Data）、インターフェース（Abstract）、実装（Concrete）、の3つのファイルに分離。

### ① `types.rs`（純粋なデータ構造）

* **書くもの**: `struct`, `enum`, `pub const`, エラー型、ビットフラグなど。
* **禁止事項**: 実行ロジック（`if`, `match` などが含まれる重い関数）の記述。他のファイル（特に同じレイヤの実装ファイル）のインポート（`use`）。
* **目的**: 依存関係の最下流に位置づけることで、**Rustのコンパイルエラーで最も発生しやすい循環参照をなくす**。

### ② `traits.rs`（抽象規約）

* **書くもの**: Rustの `pub trait`（インターフェース）の定義のみ。
* **禁止事項**: トレイトのデフォルト実装（複雑なロジックをここに書かない）。具体的な構造体（具象）への依存。
* **ルール**: スレッド間での安全な共有を保証するため、すべての抽象トレイトは原則として `Send + Sync` を自動的に要求する設計にする。
```rust
pub trait CharDevice: Send + Sync { ... }

```



### ③ `impl_xxx.rs` または `mod.rs`（具象ロジック）

* **書くもの**: `traits.rs` で定義されたトレイトの具体的な実装（`impl Trait for MyStruct`）、およびハードウェアを直接操作するようなドロドロしたロジック。
* **隠蔽ルール**: このファイルで定義される具象構造体（例: `struct SerialPort`）や内部関数は、**原則として `pub`（公開）にしない**。外の世界（システムコールなど）からは、すべて ② の `traits.rs` を通じてのみ操作させる。

## 3. ソフトウェア設計とコーディング規約（Rust/低レイヤ固有）

### 【ルール3-1】システムコール境界での依存性逆転原則（DIP）の徹底

* `syscall.rs` は、`drivers/char/serial.rs` や `drivers/fs/initrd/` といった**特定の具象クラスを直接参照してはならない**。
* システムコールがI/Oを行う際は、必ず `drivers/traits.rs`（例: `CharDevice`）や `fs/traits.rs`（例: `FileOperations`）の参照を経由して実行する。これにより、システムコール層を一切書き換えることなく、新しいデバイスをプラグインできるようになる。

### 【ルール3-2】静的ポリモーフィズムの優先（`dyn Trait` の制限）

* カーネル空間において、実行時のオーバーヘッドが大きい動的ディスパッチ（`Box<dyn Trait>` などの仮想関数テーブル参照）は、極力使用を避ける。
* 基本はRustの強みであるジェネリクス（静的ポリモーフィズム）を使い、コンパイル時に型を確定させてインライン化（最速化）を促す。
* ただし、VFS（仮想ファイルシステム）のように、実行時までファイルの種類（initrdなのか、FAT32なのか）が分からない動的なオブジェクト管理のトップレイヤに限定して、慎重に `dyn Trait`（またはポインタによる抽象化）の適用を許可する。

### 【ルール3-3】グローバル状態（`static mut` / スピンロック）の最小化

* `static mut` の直接書き換えは、レースコンディションの原因となるため**禁止**とする（アーキテクチャ依存の極一部を除く）。
* グローバルに状態を持つ必要がある場合は、必ず `spin::Mutex` などでラップする。
* **デッドロック防止ルール**: 割り込みハンドラ（`interrupts.rs`）の内部でロックを確保する可能性があるオブジェクト（例: `SERIAL1`）は、ロックを保持する時間を最小（数行以内）にし、ロックを保持したまま他の複雑な関数を呼び出してはならない。

## 4. 新規コンポーネント追加時のチェックリスト

（例：将来 、FAT32 ファイルシステム を追加する場合）

1. [ ] **ファイルを適切な場所に作成したか？**
* ➔ `src/drivers/fs/fat32/` という深いディレクトリを作る。


2. [ ] **3層分離ルールを守っているか？**
* ➔ `types.rs`（FAT32のヘッダ構造やエラー定義）、`cluster.rs`（クラスタパースロジック）、`impl_vfs.rs`（VFSとの適合層）に分ける。


3. [ ] **VFS（核心部）のコードを書き換えていないか？**
* ➔ `src/fs/vfs_core.rs` のコードには一切触れず、`fs/traits.rs` の `FileSystem` や `FileOperations` トレイトを `impl_vfs.rs` で実装するだけで完結している。


4. [ ] **依存方向は正しいか？**
* ➔ `fat32/` のコードから `src/kernel/` などの上位モジュールを逆参照（`use`）していないか確認する。


5. [ ] **境界チェックを行っているか？**
* ➔ 破損したFAT32イメージを読み込んでもカーネルパニック（Triple Faultなど）を起こさないよう、セクタ番号やクラスタ番号のバウンズチェックを徹底しているか。
